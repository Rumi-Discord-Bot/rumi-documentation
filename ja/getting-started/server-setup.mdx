毎回設定する必要はありません Rumi モジュールを一度に。まずはパーツから
コミュニティが実際に使用して、それぞれを確認し、後でさらに追加します。

## 1. 確認します Rumi 応答できる

これらをチャネルで実行します。 Rumi メッセージを読んだり送信したりできます:

```text
-ping
-help
```

どちらのコマンドも何も返さない場合は、チャネル オーバーライドを確認してください。 Rumiさんの
役割。通常の要件は、**チャンネルの表示**、**メッセージの送信**、**埋め込みです。
リンク**、**メッセージ履歴を読む**。個々の機能にはさらに多くの機能が必要になる場合があります。の
関連する `-help <command>` エントリにそれらがリストされています。

## 2. プレフィックスを確認する

このサイトの例では、 `-`。サーバーに別のプレフィックスがある場合は、次を使用します。
代わりにそのプレフィックスを使用します。スラッシュコマンド名はプレフィックスによって変更されません。

```text
-help prefix
```

「プレフィックスとスラッシュコマンド」を参照してください。](../guides/prefix-and-slash) 違いのために
2 つのコマンド スタイルの間。

## 3. 言語動作を選択する

すべてのメンバーは個人的な言語を維持できます。管理者は、
サーバーを選択していないメンバーのデフォルトサーバー。

```text
-language
-language set user de
-language set server fr
```

個人の設定はサーバーのデフォルトよりも優先されます。コマンド名はそのまま残ります
英語で; Rumi 返信とインターフェースのテキストを翻訳します。

## 4. ロールとチャネルを準備する

Rumi 続く Discordの役割階層。動く Rumiの役割はどの役割よりも優先されます
割り当て、削除、名前変更、または復元が期待されます。

便利なチェック:

```text
-permissions @Rumi
-role info Staff
-channelinfo #general
```

与えないでください Rumi 管理者はセットアップ作業を回避するためだけに使用します。権限を付与する
使用するモジュールに必要です。これにより、ボットのアクセスがわかりやすくなります。

賢明な出発点は次のとおりです。

システムを変更した後、そのステータスまたは設定コマンドを実行します。例えば、
`-levels settings`, `-economy config`、 または `-verification status` 何を示していますか Rumi
現在保存されています。

## 6.正会員としてのテスト

権限の問題はバイパスされるため、管理者が誤って隠してしまう可能性があります。
多くの Discord 制限。セットアップ後、通常のメンバーでワークフローをテストします
またはテストの役割。コマンドが実行されるチャネルと任意のチャネルの両方を確認します。
どこ Rumi 結果を送信することが期待されます。

<Tip>
何かが失敗したら実行する `-help <command>` 権限を変更する前に。それは
通常、どの権限や引数が不足しているかを推測するよりも簡単です。
</Tip>

## 非アクティブなサーバー

Rumi **サーバーが記録されないまま 30 日経過すると自動的にサーバーから離脱します
活動**。これにより、共有リソースとサービスの安定性が維持されます。何もない
サーバーが間違っているので招待できます Rumi またコミュニティのたびに
アクティブになります。
